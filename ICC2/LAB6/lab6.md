# LAB6 — 时序搭建（Timing Setup）

**实验时长**：40 分钟

## 一、学习目标

本实验为设计搭建流程的收尾环节，带你掌握 IC Compiler II 中**多工艺角多工作模式（MCMM）** 的完整配置流程。完成本实验后，你将具备以下能力：

1. 完成 MCMM 环境搭建：定义分析与优化所需的**工作模式、工艺角、时序场景**
2. 加载 MCMM 时序约束文件
3. 通过各类报告与检查命令，确认设计可进入后端实现阶段
4. 执行**零互联（ZIC）** 时序合理性校验

---

## 二、实验简介

时序搭建是整个设计搭建流程中至关重要的一步。该环节配置完成后，除非设计本身、时序约束发生变更，否则一般无需反复修改。后续工作可聚焦于布局、时钟树综合、布线等核心实现流程。

本实验主要完成以下时序相关配置与校验：

1. 搭建 MCMM 环境：定义工艺角、工作模式、时序场景，并加载对应时序约束
2. 执行多项检查，确认设计满足后端实现前置要求
3. 开展零互联时序校验，提前评估网表基础时序能力

## 六、任务 2：多工艺角多模式（MCMM）环境搭建

### 步骤 1：查看 MCMM 配置脚本

新开一个终端，进入 `lab56_setup` 目录，打开 `scripts/mcmm_ORCA_TOP.tcl` 文件。该脚本借助 Tcl 数组与循环语法，批量创建**工作模式、工艺角、时序场景**，并为每个场景加载专属时序约束。

- **问题 1**：脚本中创建的**工作模式、工艺角、时序场景**分别是什么？

### 步骤 2：加载时序约束与配置场景状态

脚本后半段会将模式、工艺角、场景对应的约束文件加载至对应对象中；最后配置每个时序场景的**分析使能状态**。可通过 `set_scenario_status` 命令手册辅助理解配置逻辑。

- **问题 2**：本实验中哪些时序场景处于激活状态？
- **问题 3**：`test.ss_125c` 场景启用了哪些分析类型？

关闭 `mcmm_ORCA_TOP.tcl`（无需保存），在 ICC II 中执行脚本完成 MCMM 初始化：

```tcl
source -echo mcmm_ORCA_TOP.tcl
```

观察终端日志：工具会提示所有场景已创建，且默认开启全部分析类型。日志中出现的**虚拟时钟、生成时钟相关告警**为正常提示，可直接忽略（虚拟时钟本身无驱动源）。

校验 `test.ss_125c` 场景的分析配置，确认与问题 3 答案一致。

### 步骤 3：清理传播时钟

时钟树综合（CTS）执行前，需要清除所有传播时钟，避免时序计算异常，执行以下循环命令：

```tcl
set cur_mode [current_mode]
foreach_in_collection mode [all_modes] {
  current_mode $mode
  remove_propagated_clocks [all_clocks]
  remove_propagated_clocks [get_ports]
  remove_propagated_clocks [get_pins -hierarchical]
}
current_mode $cur_mode
```

### 步骤 4：联动校验：模式、工艺角、场景

1. 核对当前工作模式、工艺角与时序场景是否一一对应
2. 手动切换当前模式 / 工艺角，观察时序场景是否同步变化；反之切换场景，模式与工艺角也会联动变更

> 补充说明：本设计中 `m40c` 工艺角**没有配套 `test` 工作模式的场景**。

### 步骤 5：生成模式报告 & PVT 报告

生成模式汇总报告，以表格形式查看所有场景的激活状态、分析类型（按工作模式分组展示）：

```tcl
report_mode
```

报告字段说明：

- **Current**：是否为当前生效模式
- **Default**：是否为工具默认模式（未自定义模式时，工具仅有一个名为 default 的默认模式）
- **Empty**：该模式是否未加载任何约束

生成 PVT（工艺 / 电压 / 温度）报告，检查**设计配置的 PVT 参数**与**工艺库标称 PVT 参数**是否匹配：

```tcl
view_report pvt
```

- **问题 4**：哪些工艺角出现了 PVT 参数不匹配？
- **问题 5**：不匹配的参数是工艺、电压，还是温度？
- **问题 6**：造成 PVT 不匹配的根本原因是什么？

结合工艺角名称与报错信息可判断：错误源于**用户配置的温度值**，而非工艺库本身参数。关闭 PVT 报告窗口。

### 步骤 6：修正约束文件并重新校验

1. 进入 `ORCA_TOP_constraints` 目录，编辑异常工艺角对应的约束文件：`ORCA_TOP_c_ss_m40c.tcl`
2. 修改温度配置参数，修正 PVT 不匹配问题
3. 重新执行本任务中「加载 MCMM 脚本」「清理传播时钟」「查看 PVT 报告」三步，直至 PVT 报告无异常告警

## 七、任务 3：零互联（ZIC）时序合理性校验

**零互联（Zero-Interconnect, ZIC）**：将走线寄生电阻、寄生电容强制置零，仅评估单元本身的固有延迟。用于提前判断：**纯单元延迟下，网表是否有机会满足时序要求**。若 ZIC 模式下时序仍严重违规，说明前端综合网表存在本质问题。

先查看常规模式下的时序 QoR 汇总报告：

```tcl
report_qor -summary -include setup
```

开启零互联时序计算模式，再次生成 QoR 报告：

```tcl
set_app_options -list {time.delay_calculation_style zero_interconnect}
report_qor -summary -include setup
```

**现象说明**：两种模式下时序结果差异极大。未布局的标准单元、宏单元、IO 端口之间预估走线过长，常规模式会叠加巨大的走线寄生延迟；ZIC 模式清除寄生后，时序会明显改善。

排查高扇出网络影响（本实验定义：扇出 ≥50 为高扇出网络）：

```tcl
set_app_options -list {time.high_fanout_net_pin_capacitance 0pF}
set_app_options -list {time.high_fanout_net_threshold 50}
update_timing -full
report_qor -summary -include setup
```

执行后时序结果无变化，说明**时序违规并非由高扇出网络导致**。额外现象：`func.ss_m40c`（-40℃ 慢工艺角）场景时序满足建立时间要求，其余场景出现约 2.6ns 的严重建立时间违例。

- **问题 8**：为什么同一个设计，一个场景时序达标，其余场景出现大幅时序违例？

查看最差时序路径详情，定位违规根源：

```tcl
view_report_timing -max_paths 5
```

**报告解读**：

1. 时序路径中存在大量**超大单元延迟（>1ns）**
2. 路径上的单元均为**小尺寸高阈值单元（HVT）**，属于工艺库中速度最慢的单元类型
3. 结论：这些时序关键路径**未经过综合阶段的时序优化**

**优化建议**：最优方案是在**多场景 MCMM 环境下重新做前端综合**，输出适配所有时序场景的网表；若无法重新综合，可依靠后端布局、时钟树综合、布线阶段的优化修复时序。

关闭报告窗口，退出 ICC II：

```tcl
exit
```

> 至此，**实验 6 时序搭建**全部完成。

## 补充实操复现提示

### MCMM 核心概念区分

- **模式（Mode）**：区分电路工作状态（正常功能 / 扫描测试）
- **工艺角（Corner）**：PVT 组合，模拟芯片在不同工艺偏差、电压、温度下的工作状态
- **场景（Scenario）**= 模式 + 工艺角，是 ICC II 时序分析与优化的最小执行单元

### PVT 报错处理优先级

PVT 不匹配属于**高危问题**，会导致时序计算完全失真，必须优先修正，再开展后续所有流程。

### 零互联（ZIC）使用场景

仅用于**前期网表时序摸底**，正式布局优化必须关闭 ZIC，使用真实走线寄生参数计算时序。

### 传播时钟清理

CTS（时钟树综合）前必须执行 `remove_propagated_clocks`，否则时钟延迟、时钟偏差计算会出现错误。
